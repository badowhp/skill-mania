# Technical Docs

Use this for README files, product docs, release notes, changelogs, support articles, migration guides, and developer-facing explanations.

## Priorities
1. Correctness and safety
2. Reader task completion
3. Clear structure
4. Durable terminology
5. Concise voice

## Review Checklist
- Does the reader know when to use the feature, command, or guide?
- Are prerequisites, versions, environment variables, permissions, and side effects clear?
- Are commands copyable and consistent with the repo?
- Are warnings placed before irreversible or risky actions?
- Are examples realistic, minimal, and not misleading?
- Is troubleshooting connected to observable errors?
- Are compatibility and migration notes explicit?
- Does the text avoid unsupported claims and vague hype?

## Revision Defaults
- Keep the user's technical meaning intact.
- Replace abstract benefits with concrete actions and outcomes.
- Remove assistant-shaped transitions and filler.
- Prefer short sections with direct labels over long narrative explanation.
- Use tables only when comparison or scanning is genuinely easier.
